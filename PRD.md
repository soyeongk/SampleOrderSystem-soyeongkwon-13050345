# PRD.md — 반도체 시료 생산주문관리 시스템 (SampleOrderSystem)

## 1. 배경 및 목적

가상의 반도체 회사 **S-Semi**는 다양한 반도체 시료(Sample)를 생산하여 연구소, 팹리스(Fabless) 업체,
대학 연구실 등에 납품한다.

최근 주문량이 급증하면서 엑셀·메모장 기반의 수기 관리 방식이 한계에 부딪혔다.

- 특정 주문이 처리되었는지 파악이 안 됨
- 공정 예약 후 완성 시점을 알 수 없음
- 재고가 충분한데도 불필요한 추가 공정이 돌아감

이 문제를 해결하기 위해, 시료 등록부터 주문 접수·승인·생산·출고까지 전 과정을 관리하는
**콘솔 기반 반도체 시료 생산주문관리 시스템**을 개발한다.

## 2. 사용자 및 역할

| 역할 | 설명 | 시스템에서의 행위 |
|---|---|---|
| 고객 | 시료 요청자 (연구소/팹리스/대학 연구실) | 이메일 등 외부 채널로 시료를 요청 (시스템 밖 행위) |
| 주문 담당자 | 주문서 관리자 | 고객 요청에 맞춰 시료 주문(예약) 생성 |
| 생산 담당자 | 시료 생산·승인 책임자 | 시료 등록, 주문 승인/거절, 생산 라인 관리, 출고 처리 |

이 시스템은 단일 콘솔 애플리케이션으로, 주문 담당자와 생산 담당자가 같은 콘솔을 통해
각자의 메뉴를 이용하는 것을 전제로 한다(역할별 로그인/권한 분리는 범위 밖).

## 3. 핵심 도메인 모델

### 3.1 시료 (Sample)

시스템의 가장 기본이 되는 단위. 등록된 시료만 주문 가능하다.

| 속성 | 타입 | 설명 |
|---|---|---|
| 시료 ID | string | 고유 식별자, `S-001` 형태 |
| 이름 | string | 시료명 |
| 평균 생산시간 | number | 1개 생산에 걸리는 평균 시간(분) |
| 수율 | float (0~1) | 정상 시료 수 / 총 생산 시료 수 (예: 100개 중 90개 정상 = 0.9) |
| 현재 재고 | int | 조회 시 표시되는 현재 보유 수량 |

### 3.2 주문 (Order)

| 속성 | 타입 | 설명 |
|---|---|---|
| 주문번호 | string | `ORD-YYYYMMDD-HHMM-NN` 형태, 유일해야 함 |
| 시료 ID | string | 주문 대상 시료 |
| 고객명 | string | 주문 고객 |
| 주문 수량 | int | 요청 수량 |
| 상태 | enum | 아래 상태표 참조, 계속 갱신됨 |

**주문 상태**

| 상태 | 의미 |
|---|---|
| RESERVED | 주문 접수 |
| REJECTED | 주문 거절 (정상 흐름 밖의 상태, 모니터링 집계에서 제외) |
| PRODUCING | 주문 승인 완료 + 재고 부족으로 생산 중 |
| CONFIRMED | 주문 승인 완료 + 출고 대기 중 |
| RELEASED | 출고 완료 |

**상태 전이**

```
RESERVED --(거절)--------------------------------------> REJECTED
RESERVED --(승인, 재고 충분)--> CONFIRMED --(출고 처리)--> RELEASED
RESERVED --(승인, 재고 부족)--> PRODUCING --(생산 완료)--> CONFIRMED --(출고 처리)--> RELEASED
```

## 4. 기능 요구사항

### 4.1 메인 메뉴
- 시료 관리 / 시료 주문 / 주문 승인·거절 / 모니터링 / 생산 라인 조회 / 출고 처리로 이동하는 진입점
- 전체 시료 요약 정보(등록 시료 수, 총 재고, 전체 주문 수, 생산 라인 대기 건수 등) 표시
- 화면 레이아웃은 자유롭게 결정 가능 (참고 UI는 예시일 뿐 강제 사항 아님)

### 4.2 시료 관리
| 기능 | 입력 | 동작 |
|---|---|---|
| 시료 등록 | 이름, 평균 생산시간, 수율 | 시료 ID 자동 채번 후 신규 등록 |
| 시료 조회 | - | 등록된 전체 시료 + 현재 재고 목록 표시 |
| 시료 검색 | 이름 등 검색어 | 조건에 맞는 시료만 필터링해 표시 |

### 4.3 시료 주문 (예약)
- 입력: 시료 ID, 고객명, 주문 수량
- 처리: 주문번호(`ORD-YYYYMMDD-HHMM-NN`) 자동 생성, 상태 `RESERVED`로 신규 주문 생성
- 존재하지 않는 시료 ID로는 주문할 수 없음

### 4.4 주문 승인/거절
- `RESERVED` 상태 주문 목록을 조회하여 특정 주문 선택
- **승인**
  - 재고 충분 → 즉시 `CONFIRMED`
  - 재고 부족 → 생산 큐에 자동 등록 + 상태 `PRODUCING`
- **거절** → 즉시 `REJECTED`

### 4.5 모니터링
- **주문량 확인**: 상태별(RESERVED/PRODUCING/CONFIRMED/RELEASED) 건수 집계. `REJECTED`는 집계 제외
- **재고량 확인**: 시료별 재고 수량과 상태
  - 여유: 주문 대비 재고 충분
  - 부족: 주문 대비 재고 부족
  - 고갈: 재고 0

### 4.6 생산 라인
- 승인 시 재고가 부족한 주문은 생산 큐(FIFO)에 등록되어 순서대로 처리
- **실 생산량** = `ceil(부족분 / 수율)`
- **총 생산 시간** = `평균 생산시간 * 실 생산량`
- 총 생산 시간이 모두 경과해야 주문 상태가 `PRODUCING → CONFIRMED`로 전환됨 (부분 생산 완료 시점에 조기 전환되지 않음)
- 현재 생산 중인 시료/주문 정보와 생산 큐 대기 목록을 조회 가능

### 4.7 출고 처리
- `CONFIRMED` 상태 주문만 출고 대상으로 노출
- 출고 실행 시 상태 `RELEASED`로 전환

## 5. 비기능 요구사항

| 항목 | 요구사항 |
|---|---|
| 실행 형태 | 콘솔(CLI) 애플리케이션 |
| 언어 | Python |
| 데이터 저장 | JSON 파일 기반 영속성. Repository 계층이 파일 read/write를 캡슐화하여 Model이 저장 방식을 알지 못하도록 분리 |
| 아키텍처 | MVC (Model / View / Controller) 계층 분리 |
| 데이터 유지 | 애플리케이션을 재실행해도 기존 데이터가 유지되어야 함 |
| 코드 품질 | Clean Code, 테스트 코드 존재 (Test), 의미 있는 Commit 이력 |

## 6. 범위 밖 (Out of Scope)

- 웹/GUI 화면 (콘솔 UI로 한정)
- 사용자 인증/권한 분리 (역할 구분은 메뉴 단위로만 존재)
- 관계형 DB, 외부 DB 서버 연동 (JSON 파일로 대체)
- 다중 생산 라인 동시 운영 (하나의 생산 라인이 시료를 하나씩 순차 생산)
- 실시간 알림/이메일 발송 (고객의 시료 요청은 시스템 밖에서 이루어지는 것으로 가정)

## 7. 성공 기준 (Definition of Done)

- [ ] 시료 등록 → 주문 예약 → 승인(재고 충분/부족 양쪽 경로) → (필요 시 생산 완료) → 출고까지 전체 흐름이 정상 동작
- [ ] 애플리케이션 재실행 후에도 이전 데이터가 유지됨
- [ ] 모니터링에서 상태별 주문 수·재고 상태(여유/부족/고갈)가 정확히 집계됨
- [ ] 생산 큐가 FIFO 순서를 지키며, 총 생산 시간 경과 전에는 상태가 조기 전환되지 않음
- [ ] 주요 기능에 대한 테스트 코드가 존재하고 통과함
- [ ] MVC 계층 분리가 유지됨 (Controller ↔ Model/View 단방향 의존)
